> آخر تحديث: 2026-07-01 · الفرع: `vue-test` · الحالة: التطبيق مكتمل وظيفيّاً كنموذج غنيّ ببيانات محفوظة محليّاً (IndexedDB).
> هذه الخطة تغطّي **كل ما تبقّى قبل الإنتاج باستثناء الباك-اند/الـAPI**. موضوع **اللغة/التعريب (i18n) مؤجَّل** بقرار المستخدم — لا يُنفَّذ الآن.

## الوضع الحالي (مرجع سريع)

- **مكتمل**: كل الموديولات (لوحة تحكم، مشاريع/أعضاء، مالية/سجل، ذمم، التزامات، أصول + ضمانات فرعية، متابعات + مساعد ذكي، طلبات (13 نوعاً)، مستندات + مركز إجراءات كامل، استبيانات، إشعارات، تدقيق، مهام، إعدادات).
- **نظام تصميم** جاهز: رموز (tokens)، مكوّنات نواة (`src/components/base/`: BaseButton/Field/Input/Select/Textarea/EmptyState/StatusBadge/Tooltip/ToastHost/CommandPalette)، Toasts (`useToast`)، ترقية ModalShell (حركة/Esc/حبس تركيز)، وصول أساسي، كثافة (comfortable/compact)، لوحة أوامر ⌘K. (راجع ذاكرة `mazeen-design-system`.)
- **مؤجَّل صراحةً**: التعريب الكامل / ثنائية اللغة.
- **خارج نطاق هذه الخطة**: الباك-اند، الـAPI، الذكاء الاصطناعي الحقيقي (كلها تعتمد خادماً).

## 🔴 المرحلة أ — البنية الحرجة (أعلى أولوية)

### أ-1 · إطار الاختبارات + التغطية ✅ (مُنجَز 2026-07-01)
> **النتيجة**: Vitest + @vue/test-utils + jsdom + تغطية v8. **147 اختباراً** أخضر عبر 19 ملفاً. تغطية **66.6% أسطر / 89.3% فروع** (تجاوز هدف ≥60%). المنطق الأساسي: helpers 100% · FinanceStore 96% · RequestsStore 91% · TrackingsStore 95% · CommitmentsStore 84% · لوحة التحكم 92%. سكربتات `test`/`test:watch`/`test:coverage`. استُبعدت ملفات IO (export/persist/documents) من القياس (تُغطّى عبر E2E/يدويّاً لاحقاً).
- **الهدف**: شبكة أمان قبل أي تغيير كبير. لا توجد اختبارات إطلاقاً حاليّاً.
- **المهام**:
  - إعداد **Vitest + @vue/test-utils + jsdom** (+ سكربت `test`/`test:coverage` في package.json).
  - اختبارات وحدة للمنطق النقي أولاً (أعلى عائد): `helpers/calc.ts`، `helpers/txAnalysis.ts`، `helpers/date.ts`، `helpers/format.ts`، و`composables/useHealthCheck`, `useDashboard`, `useTasks`, `trackingsAI`, `docAI`.
  - اختبارات متاجر Pinia للتدفّقات الحرجة: `RequestsStore.decide` (اعتماد → عملية مالية بالاتجاه الصحيح)، `CommitmentsStore.payCommitment`، `AssetsStore.addMaintenance/addWarranty/linkSubWarranty`، `TrackingsStore.renewTracking(feeAsExpense)`، `FinanceStore.saveTransaction` (تحويل → قيدان).
  - اختبارات مكوّنات لمكوّنات النواة (`BaseButton/BaseField/…`) وModalShell (Esc/حبس التركيز).
  - (لاحقاً) **Playwright** لمسارات E2E: تسجيل دخول → لوحة → إنشاء طلب → اعتماد → ظهوره في المالية.
- **قبول**: `npm run test` أخضر، تغطية ≥ 60% للمنطق (helpers/stores/composables)، وتشغيلها في CI.

### أ-2 · المصادقة والصلاحيات (فرونت-اند)
- **الهدف**: استبدال الدخول الصوري بتدفّق حقيقي جاهز لربط الباك-اند. حاليّاً `LoginPage.submit()` يفعل `router.push` فقط، والحارس يُتجاوَز (`REQUIRE_AUTH=false`).
- **المهام**:
  - `AuthStore`: حالة مستخدم + توكن، `login()/logout()`، `isAuthenticated`, `currentUser`, `role`. تخزين التوكن (مع مراعاة الأمان لاحقاً) + استرجاع الجلسة عند الإقلاع.
  - `LoginPage` يستدعي `AuthStore.login` (واجهة وهمية الآن: مستخدم/كلمة تجريبية أو أي إدخال → مستخدم افتراضي) مع حالات خطأ/تحميل.
  - تفعيل حارس المسار فعليّاً (بمفتاح تهيئة يبقى «وهمي» حتى الباك-اند)، + إعادة توجيه `redirect` بعد الدخول، + تسجيل خروج من الشريط العلوي.
  - **تقييد الواجهة حسب الدور** (مالك/مدير/عضو/مشاهد الموجودة في نموذج العضو): إخفاء/تعطيل أزرار الإنشاء/الحذف/الاعتماد حسب الصلاحية عبر توجيه `v-if`/directive مركزي (`useCan(permission)`).
- **قبول**: لا وصول للأقسام بلا «دخول»؛ زر خروج يعمل؛ عناصر واجهة تختفي/تتعطّل حسب الدور؛ الجلسة تدوم عبر التحديث.

## 🟠 المرحلة ب — اكتمال المنتج

### ب-1 · حالات الذكاء الاصطناعي (الجزء الأمامي)
- **الهدف**: تجهيز `docAI`/`trackingsAI` (محاكاة الآن) لاستبدالها لاحقاً بخادم دون تغيير الواجهة.
- **المهام**: عزل الاستدعاء خلف واجهة `aiService` واحدة (تُرجع Promise)، إضافة حالات **تحميل/خطأ/إعادة محاولة** موحّدة، ودعم عرض تدريجي للنتائج. الإبقاء على المحاكاة كـ«مزوّد» افتراضي.
- **قبول**: تبديل المزوّد (محاكاة ↔ خادم) نقطة واحدة؛ أخطاء التحليل تظهر بأناقة مع إعادة محاولة.

### ب-2 · تدقيق اكتمال الشاشات (CRUD والـplaceholders)
- **المهام**: إزالة/استبدال مسار `ComingSoonPage` المتبقّي؛ جرد كل موديول للتأكد من اكتمال (إنشاء/عرض/تعديل/حذف/إجراءات) دون ثغرات؛ توحيد أنماط التفاصيل/الهَب.
- **قبول**: لا شاشة «قريباً»؛ جدول تغطية يؤكّد اكتمال كل موديول.

### ب-3 · تبنّي نظام التصميم (إزالة الدَّين)
- **الهدف**: توحيد المظهر وتقليل التكرار. المكوّنات موجودة لكنها غير معتمدة على نطاق واسع.
- **المهام**: ترحيل تدريجي (موديول-موديول) لـ:
  - الأزرار المبعثرة `.app-btn` → `BaseButton`.
  - حقول الإدخال المكرّرة (≈22 ملفاً) → `BaseField` + `BaseInput/Select/Textarea`.
  - حالات الفراغ اليدوية → `EmptyState`؛ شارات الحالة → `StatusBadge`.
- **قبول**: صفر تعريفات أزرار محلية؛ الحقول موحّدة؛ فحص أنواع وبناء نظيف بعد كل موديول.
